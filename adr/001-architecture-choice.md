# ADR-001: Выбор архитектурного стиля
* Дата: 04.02.2026
* Статус: Принято
* Контекст:  
Компания имеет большой мультиязычный штат разработчиков, которые пишут на разных языках. 90% систем находятся в облаке, вендор лока нет. Приложение должно поддерживать массовые соревнования и интеграцию разнородных функций(соцсеть, трекинг, аналитика)  
* Опции:  
- Монолитная архитектура (Layered Monolith)  
​Описание: Вся логика (соцсеть, аналитика, уведомления) в одном деплой-юните.  
​Плюсы: Простота отладки на старте, транзакционная целостность.  
​Минусы: Невозможность использования разных языков программирования (противоречит). Сложность масштабирования «тяжелых» модулей (Analytics) отдельно от легких (Notification). Риск отказа всей системы при ошибке в одном модуле.  
- ​Сервис-ориентированная архитектура (SOA / Microservices)  
​Описание: Разделение системы на независимые сервисы (Identity, Social, Tracking, Analytics, Promo), общающиеся по сети.  
​Плюсы: Возможность использовать Polyglot-стек (Java для Enterprise, Python для Analytics, Go для Highload). Независимое масштабирование. Изоляция сбоев.  
​Минусы: Сложность инфраструктуры, распределенные данные.  
* Принятое решение:  
Микросервисы позволят командам выбирать стек(Go, Java, Python, JavaScript...) под конкретную задачу.  
* Последствия:
- Масштабируемость - разделение функционала (соцсеть vs трекинг) позволяет независимо масштабировать нагруженные части(например обработку телеметрии)
- Наблюдаемость - плохая, из-за того что в такой архитектуре сложнее отследить какой из узлов оказался причиной сбоя
- Deployability - независимое обновление компонентов
- Testability - хорошая
- Простота в обслуживании
